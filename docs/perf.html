Full changelog

Baseline
  • Full-resolution frame sent to the VLM every call
  • A fresh DB connection + query per cache lookup
  • No region cache — every whiteboard tap re-ran the model
PR #4
  • Downscale frame to ≤768 px before the VLM call
  • In-process LRU + persistent cache connection
  • Cache /api/region by (video, frame, bbox)
  • Single-writer telemetry queue + the live ?view=perf dashboard
PR #5
  • vLLM (vllm-mlx) backend alongside LM Studio — parity on the VLM call (~1.33 s vs ~1.40 s)
  • Client + server request timeouts & a warmup ping
  • Kills the disconnected-client pileup that spiked tail latency to 100 s+ under load

Which vLLM-mlx features we use — and which we skip

vllm-mlx 0.4.0 · an MLX reimpl, not upstream vLLM
Using6 features
  • Continuous in-flight batching — --continuous-batching runs many requests at once; the offline sweep fans across it for the throughput win below.
  • Structured outputs — response_format json_schema; the 4B emits schema-valid concept JSON on the vision path, verified end-to-end.
  • OpenAI-compatible + base64 images — drop-in beside LM Studio; one client path, no rewrite.
  • Request timeouts + warmup ping — client KEDU_TIMEOUT (no retries) + server --timeout; kills the disconnected-client / cold-kernel pileup.
  • Frame downscale ≤768px + bounded max_tokens — fewer visual tokens per call → faster prefill, more sequences fit the batch.
  • Opt-in KV-cache quant · --max-num-seqs — headroom to fit more concurrent image sequences when memory allows.
Skippedwhy not
  • PagedAttention — not present in this MLX port (only an experimental block cache). Nothing to adopt.
  • Prefix cache for our prompt — the prefix caches are text-only; an image skips them, so the fixed system prompt re-prefills regardless. We cut tokens instead.
  • Speculative decoding (MLLM draft) — SimpleEngine-only, mutually exclusive with continuous batching. We keep batching.
  • Default SimpleEngine — serializes and 503s the 2nd concurrent request — the opposite of what the sweep needs.
  • Hidden states / logprobs — absent on the OpenAI endpoints; the pipeline doesn't need them.
  • Tool calling — supported, but concept extraction is single-shot structured output; no tools.
