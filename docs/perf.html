Time to first widget, release over release.

The mint hot path went from over the 5-second budget to well under 1.5 s — and each drop is tagged with the change that caused it. The latest release (PR #6) swaps the whole model server to vllm-metal — upstream vLLM with real PagedAttention; that latency isn't re-benchmarked here (the host was memory-bound — see method), so the charts run through PR #5 and the migration is logged below.

Full changelog

Baseline
  • Full-resolution frame sent to the VLM every call
  • A fresh DB connection + query per cache lookup
  • No region cache — every whiteboard tap re-ran the model
PR #4
  • Downscale frame to ≤768 px before the VLM call
  • In-process LRU + persistent cache connection
  • Cache /api/region by (video, frame, bbox)
  • Single-writer telemetry queue + the live ?view=perf dashboard
PR #5
  • vLLM (vllm-mlx) backend alongside LM Studio — parity on the VLM call (~1.33 s vs ~1.40 s)
  • Client + server request timeouts & a warmup ping
  • Kills the disconnected-client pileup that spiked tail latency to 100 s+ under load
PR #6 · vLLM Metal
  • Native vllm-metal — upstream vLLM 0.25.1 + a Metal backend, replacing the third-party vllm-mlx port
  • PagedAttention (default paged-varlen Metal kernel) + prefix caching + real continuous batching — the features the MLX port couldn't offer
  • Brain moved to DeepSeek-R1-Distill-Qwen-7B on the same server (Nemotron's MoE isn't supported); vision keeps the same Qwen3-VL-4B
  • Client unchanged — same OpenAI path; only the server + serve-vllm-metal.sh are new

Which vLLM features we use on vllm-metal — and which we skip

vllm-metal 0.3.0 · upstream vLLM 0.25.1 + Metal plugin
Usingnow on Metal
  • PagedAttention — the default paged-varlen Metal kernel (server log: "Native paged-attention Metal kernels loaded"). The MLX port never had this — here it is the attention backend, not an add-on. This is the migration's headline.
  • Prefix caching — --enable-prefix-caching, default-on for the paged KV cache; the fixed genre system-prompt prefix is reused across calls instead of re-prefilled.
  • Continuous batching + chunked prefill — upstream vLLM's real scheduler (--max-num-seqs), not an MLX reimpl; chunked prefill rides the unified varlen kernel.
  • Structured outputs — response_format json_schema; verified schema-valid concept JSON on the Qwen3-VL vision path, on Metal.
  • OpenAI-compatible + base64 images — the client is unchanged from the LM Studio / vllm-mlx path; only the server swapped.
  • Request timeouts + warmup ping — client KEDU_TIMEOUT + a warmup that absorbs the cold Metal kernel compile.
Skippedavailable, not on
  • Speculative decoding — now available on vllm-metal (mtp / draft / ngram), unlike the MLX port — but it's greedy-only and the vision path samples at temp 0.1. Left off.
  • TurboQuant KV compression — supported via --additional-config (needs paged attention, which we now have); experimental, so off until A/B'd.
  • --kv-cache-dtype fp8 — CUDA/ROCm-only; rejected on Metal at config time. TurboQuant is the Metal route if we want KV compression.
  • Tensor parallelism (TP>1) — only TP=1 is validated on Metal (single device); a 4B/7B doesn't need it.
  • Hidden states / logprobs — the pipeline doesn't need them.
  • Tool calling — concept extraction is single-shot structured output; no tools.

Method. PR #4 deltas are its measured before/after on the Excel lecture (qwen3-vl-4b). PR #5 figures are a same-method benchmark on video 5IgOP7Lpk5g — identical model, direct-to-server, warmed, sequential — giving ~1.33 s (vLLM) vs ~1.40 s (LM Studio) on a clean host. Warm-hit and region wins are cache-driven and server-independent, so PR #5/#6 hold them. PR #6 (vllm-metal) is an architectural migration — PagedAttention + prefix caching + real batching, each validated end-to-end — but its single-request latency is not re-benchmarked here: the measuring host was memory-bound (heavy swap), which confounds absolute numbers, and PagedAttention's win is a throughput effect under concurrency, not single-request cold-miss (prefill+decode-bound). A clean-host re-benchmark is the open follow-up.
